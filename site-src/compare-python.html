pymodelet vs the SQL-first field

Every SQL-first tool in the Python ecosystem takes care of the read side. pymodelet makes even the writes effortless.

pymodelet is currently the only package where you run SQL to get a list of entities, save that same list right back — and insert, update or delete is decided automatically.

Here is what still sets pymodelet apart.

The field

Four SQL-first packages at a glance

pymodelet

Reads and writes, both answered

Plain SQL in, typed entities out — on any DB-API 2.0 driver, zero dependencies. And the part nobody else does: saveOrUpdate generates every INSERT / UPDATE / DELETE from the entity, writes the key back, stamps audit fields.

psycopg 3 · row_factory

Beautiful reads, PostgreSQL only

class_row maps SELECTs into dataclasses with zero extra packages — the cleanest read path there is, if you are on PostgreSQL. Every write statement is still yours to compose, and the key comes back only if you remember RETURNING.

aiosql

SQL files as functions

Queries live in .sql files and become callable functions — the most elegant way to manage the read side. But writes are hand-authored one by one, and rows come back as rows: turning them into typed entities is your job.

SQLAlchemy Core + Pydantic

Solid plumbing, manual mapping

text() queries over a battle-tested pool, entities validated by Pydantic. But the mapping is manual per query — UserEntity(**row._mapping) — and the write side is exactly where you left it: unwritten.

The decisive difference

pymodelet: effortless reads and writes, readable code, fewer AI tokens

Picture a routine data-CRUD scenario: create a user, save it, then fix the email later. The other packages make you author both SQL statements; pymodelet makes them disappear — you and your AI stay focused on business logic instead of generating pointless tokens.

aiosql — users.sql + Pythonevery statement, by hand
-- users.sql — you write and maintain each one
-- name: insert_user!
INSERT INTO users (name, email) VALUES (:name, :email);

-- name: update_user_email!
UPDATE users SET email = :email WHERE id = :id;

# Python
queries.insert_user(conn, name="Alice", email="alice@test.com")
queries.update_user_email(conn, id=1, email="alice@corp.com")
pymodeletzero statements
user = User(name="Alice", email="alice@test.com")
model.save(user)          # INSERT — id written back

user.email = "alice@corp.com"
model.save(user)          # UPDATE — same call, saveOrUpdate

# delete? user.txn_mode = TxnMode.DELETE; model.save(user)

Add a column: in aiosql that touches the table, the INSERT, the UPDATE and every call site that names the fields. In pymodelet it touches the table and one dataclass field — the SQL regenerates itself.
