The Philosophy

The opposite trade-off to the ORM wave

In 2006, Hibernate asked engineers to learn HQL, session lifecycles and mapping rituals before writing a single productive line. Modelet does the reverse: respect SQL, and only take over the tedious part.

SELECTQueries → SQL

SELECTs are written in the ANSI SQL engineers already know — subqueries and joins included. The framework only carries each row back into an object. No new query language.

WRITEWrites → framework

What's mechanical and error-prone is enumerating columns for INSERT / UPDATE / DELETE. Modelet generates them from the entity's TxnMode state, writes the key back, and stamps audit fields.

Built for scale

Deep pagination without the memory bill

Whether the resultset is 100 rows or 500,000, only the rows of the page you asked for become entities. Reaching the very last page never loads the whole set into your application.

LIMIT / OFFSET is pushed down to the database; the application tier maps only page-size rows. Memory and mapping cost stay flat — O(page), not O(total) — from page 1 to page 50,000. The old scan-everything-into-memory approach is gone.

How it flows

From SQL to entity and back — automatically

Write SQL, get a list of entities. A web form binds user input into the entity through Expression Language. The middle tier just calls saveOrUpdate on that same entity — Modelet turns it back into SQL.

Entities are declared with JPA-compatible annotations — @Table @Id @Column @Transient @Enumerated @GeneratedValue — so they read the same as any Jakarta Persistence entity. Only the vocabulary is borrowed; the engine stays SQL-first.

Quick Start

One example, end to end

A classic one-to-many: an Author has many Books. The foreign key is the relationship — no @OneToMany, no lazy loading.

Class diagram

1Define entities with JPA-compatible annotations

models.pyPython 3.10+
@Table("author")
@dataclass
class Author(Entity):
    name: str | None = None

@Table("book")
@dataclass
class Book(Entity):
    author_id: int | None = None          # FK = the relationship
    title: str | None = Column("bookName")
    price: Decimal | None = None

2Query with the SQL you know

query.pyfind → list
# plain ANSI SQL in, a list of entities out
books = model.find(
    "select * from book where author_id = ? order by id",
    [author_id], Book)

3saveOrUpdate — the framework writes the SQL

save.pysave
book = Book(author_id=author.id,
            title="pymodelet in Action")
model.save(book)    # new entity → INSERT, id written back

book.price = Decimal("499")
model.save(book)    # loaded entity → UPDATE. same call.
