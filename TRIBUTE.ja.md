## Matt へ、そして 2006 年のあの決断へ

2006 年、Java の世界は ORM の波に呑まれていました。Hibernate はその時代の
答えでしたが、その答えには高くつく問題が潜んでいました。エンジニアは、
生産的な最初の一行を書く前に、HQL、セッションのライフサイクル、遅延ロード
の落とし穴、マッピングの作法を習得しなければならなかったのです。SQL に
精通したエンジニアでさえ、ORM の前では初心者のように立ち尽くしました。

その年、Matt は的確な決断を下しました。**ER モデルと SQL は基本である ——
技術の移り変わりがその価値を消すことはなく、ビジネスロジックの最も明快な
表現であり続ける。**

彼の洞察は次のような概念に凝縮できます —— ビジネスロジックのクエリ条件は自分で書くべきだが、オブジェクトの永続化はフレームワークに任せ、追加・更新・削除を自動で判定させるべきだ。SELECT はエンジニアがすでに知っている ANSI SQL で書けばよく、サブクエリが書ければ初日から始められます。本当に面倒で機械的で間違えやすいのは、INSERT・UPDATE・DELETE のカラム列挙です —— それこそフレームワークが引き受けるべき部分。こうして Modelet は生まれました。1 時間で読み切れるほど小さなフレームワーク、ほとんど存在しない学習曲線、そして期日どおりに納品し続けたいくつものチーム。

20 年が過ぎ、今日の Python コミュニティは「SQL-first」をまるで新しいもののように再発見しています。Matt は 2006 年にそれを書き記していました。

pymodelet はこの理念の Python での再生です。Entity、TxnMode、Model、PageContainer、DataRoller —— どの名前も Java 版からそのまま受け継がれています。技術が次々と移り変わっても、色あせないのは、20 年前にビジネスの視点から下されたこの決断です。
