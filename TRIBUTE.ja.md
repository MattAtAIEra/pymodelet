## Matt へ、そして 2006 年のあの決断へ

2006 年、Java の世界は ORM の波に呑まれていました。Hibernate はその時代の
答えでしたが、その答えには高くつく問題が潜んでいました。エンジニアは、
生産的な最初の一行を書く前に、HQL、セッションのライフサイクル、遅延ロード
の落とし穴、マッピングの作法を習得しなければならなかったのです。SQL に
精通したエンジニアでさえ、ORM の前では初心者のように立ち尽くしました。

その年、Matt は的確な決断を下しました。**ER モデルと SQL は基本である ——
技術の移り変わりがその価値を消すことはなく、ビジネスロジックの最も明快な
表現であり続ける。**

彼の洞察は一文に凝縮できます —— クエリは SQL に、書き込みはフレームワークに。
SELECT はエンジニアがすでに知っている ANSI SQL で書けばよく、サブクエリが
書ければ初日から始められます。本当に面倒で機械的で間違えやすいのは、
INSERT・UPDATE・DELETE のカラム列挙です —— それこそフレームワークが引き
受けるべき部分。こうして Modelet は生まれました。半日で読み切れるほど
小さなフレームワーク、ほとんど存在しない学習曲線、そして期日どおりに
納品し続けたいくつものチーム。

20 年が過ぎ、その判断は時に耐えました。今日の Python コミュニティは
「SQL-first」をまるで新しいもののように再発見しています。Matt は 2006 年に
それを書き記していました。

pymodelet はこの理念の Python での再生です。Entity、TxnMode、Model、
PageContainer —— どの名前も Java 版からそのまま受け継がれています。改良の
必要はなく、ただ翻訳されるべきものだったからです。このフレームワークの
おかげで初日から動くコードを書けたすべてのエンジニアへ —— その学習曲線は、
2006 年に誰かがあなたのために平らにしておいたものです。
