pymodelet vs SQL-first 勢

Python エコシステムの SQL-first ツールはどれも読み取り側の面倒を見てくれます。pymodelet は書き込みまで楽にしてくれます。

SQL を投げて list of entity を受け取り、その list をそのまま保存すれば、追加・更新・削除は自動判定 —— それができるのは、現時点で pymodelet だけです。

それでも pymodelet が際立つ理由を示します。

競合たち

4 つの SQL-first パッケージの紹介

pymodelet

読みも書きも、答えあり

素の SQL を入れ、型付きエンティティが出てくる —— どの DB-API 2.0 ドライバでも、依存ゼロで。そして誰もやらなかった部分:saveOrUpdate がエンティティから INSERT / UPDATE / DELETE を全て生成し、キーを書き戻し、監査項目を付与します。

psycopg 3 · row_factory

美しい読み取り、ただし PostgreSQL 限定

class_row は SELECT を追加パッケージなしで dataclass に写します —— PostgreSQL なら最も綺麗な読み取り経路。しかし書き込み文はすべて自作、キーの取得も RETURNING を忘れなければ、の話です。

aiosql

SQL ファイルが関数になる

クエリは .sql ファイルに住み、呼び出せる関数になります —— 読み取り側の管理として最もエレガント。しかし書き込みは一文ずつ手書きで、返るのは row のまま:型付きエンティティへの変換はあなたの仕事です。

SQLAlchemy Core + Pydantic

配管は堅牢、マッピングは手動

実績あるプール上で text() クエリ、エンティティは Pydantic が検証。しかしマッピングはクエリごとに手動 —— UserEntity(**row._mapping) —— そして書き込み側は手つかずのまま残ります。

決定的な違い

pymodelet なら読み書きが楽、コードは読みやすく、AI のトークンも節約

ありふれたデータ CRUD の場面を想像してください:ユーザーを登録して保存し、あとからメールアドレスを修正する。他のパッケージでは二つの SQL 文を自分で書くことになりますが、pymodelet ならそれらは消えてなくなります。あなたと AI は業務ロジックに集中でき、無駄なトークン生成も省けます。

aiosql — users.sql + Pythonevery statement, by hand
-- users.sql — you write and maintain each one
-- name: insert_user!
INSERT INTO users (name, email) VALUES (:name, :email);

-- name: update_user_email!
UPDATE users SET email = :email WHERE id = :id;

# Python
queries.insert_user(conn, name="Alice", email="alice@test.com")
queries.update_user_email(conn, id=1, email="alice@corp.com")
pymodeletzero statements
user = User(name="Alice", email="alice@test.com")
model.save(user)          # INSERT — id written back

user.email = "alice@corp.com"
model.save(user)          # UPDATE — same call, saveOrUpdate

# delete? user.txn_mode = TxnMode.DELETE; model.save(user)

カラムを一つ追加すると:aiosql ではテーブル、INSERT、UPDATE、そしてフィールド名を列挙する全呼び出し箇所に手が入ります。pymodelet ならテーブルと dataclass のフィールド一つ —— SQL は自動的に生え変わります。
