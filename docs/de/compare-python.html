pymodelet vs. das SQL-first-Feld

Jedes SQL-first-Tool im Python-Ökosystem kümmert sich um die Leseseite. Mit pymodelet werden selbst die Schreibvorgänge mühelos.

pymodelet ist derzeit das einzige Paket, bei dem du SQL absetzt, eine Liste von Entitäten erhältst und dieselbe Liste direkt zurückspeicherst — Insert, Update oder Delete wird automatisch entschieden.

Hier ist, was pymodelet weiterhin abhebt.

Das Feld

Vier SQL-first-Pakete im Überblick

pymodelet

Lesen und Schreiben, beides beantwortet

Pures SQL rein, typisierte Entitäten raus — auf jedem DB-API-2.0-Treiber, null Abhängigkeiten. Und der Teil, den sonst niemand macht: saveOrUpdate generiert jedes INSERT / UPDATE / DELETE aus der Entität, schreibt den Schlüssel zurück, setzt Audit-Felder.

psycopg 3 · row_factory

Schönes Lesen, nur PostgreSQL

class_row mappt SELECTs ohne Zusatzpakete in Dataclasses — der sauberste Lesepfad, wenn du auf PostgreSQL bist. Jedes Schreib-Statement bleibt deine Aufgabe, und den Schlüssel gibt es nur mit RETURNING zurück.

aiosql

SQL-Dateien als Funktionen

Abfragen wohnen in .sql-Dateien und werden aufrufbare Funktionen — die eleganteste Verwaltung der Leseseite. Aber Schreibvorgänge werden Stück für Stück von Hand verfasst, und Zeilen kommen als Zeilen zurück: die Umwandlung in typisierte Entitäten ist dein Job.

SQLAlchemy Core + Pydantic

Solide Leitungen, manuelles Mapping

text()-Abfragen über einem bewährten Pool, Entitäten von Pydantic validiert. Aber das Mapping ist pro Abfrage manuell — UserEntity(**row._mapping) — und die Schreibseite bleibt genau da, wo du sie gelassen hast: ungeschrieben.

Der entscheidende Unterschied

pymodelet: Lesen und Schreiben mühelos, lesbarer Code, weniger KI-Tokens

Stell dir ein alltägliches Daten-CRUD-Szenario vor: einen Benutzer anlegen, speichern, später die E-Mail korrigieren. Bei den anderen Paketen schreibst du beide SQL-Statements selbst; pymodelet lässt sie verschwinden — du und deine KI bleibt bei der Geschäftslogik, statt sinnlose Tokens zu erzeugen.

aiosql — users.sql + Pythonevery statement, by hand
-- users.sql — you write and maintain each one
-- name: insert_user!
INSERT INTO users (name, email) VALUES (:name, :email);

-- name: update_user_email!
UPDATE users SET email = :email WHERE id = :id;

# Python
queries.insert_user(conn, name="Alice", email="alice@test.com")
queries.update_user_email(conn, id=1, email="alice@corp.com")
pymodeletzero statements
user = User(name="Alice", email="alice@test.com")
model.save(user)          # INSERT — id written back

user.email = "alice@corp.com"
model.save(user)          # UPDATE — same call, saveOrUpdate

# delete? user.txn_mode = TxnMode.DELETE; model.save(user)

Eine neue Spalte: In aiosql berührt sie die Tabelle, das INSERT, das UPDATE und jede Aufrufstelle, die Felder benennt. In pymodelet nur die Tabelle und ein Dataclass-Feld — das SQL erzeugt sich neu.

Loslegen

Lass auch deine KI die Befriedigung von sauberem Code erleben — lesbar, wartbar, nicht bloß weniger Tokens.
