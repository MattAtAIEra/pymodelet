쿼리는 SQL에게, 쓰기는 프레임워크에게.

이미 아는 ANSI SQL로 쿼리를 작성하고, 번거로운 INSERT / UPDATE / DELETE는 프레임워크에 맡기세요. 거의 존재하지 않는 학습 곡선 —— 20년째 그대로입니다.

철학

ORM 물결과 정반대의 선택

2006년, Hibernate는 생산적인 한 줄을 쓰기도 전에 HQL, 세션 생명주기, 매핑 관례부터 익히길 요구했습니다. Modelet은 반대로 합니다. SQL을 존중하고, 번거로운 부분만 대신합니다.

SELECT쿼리 → SQL

SELECT는 엔지니어가 이미 아는 ANSI SQL로 작성합니다 —— 서브쿼리와 조인 포함. 프레임워크는 각 행을 객체로 되돌릴 뿐, 새로운 쿼리 언어는 없습니다.

WRITE쓰기 → 프레임워크

기계적이고 실수하기 쉬운 것은 INSERT / UPDATE / DELETE의 컬럼 나열입니다. Modelet은 엔티티의 TxnMode 상태로부터 이를 생성하고, 키를 되써 주고, 감사 필드를 채웁니다.

대규모를 위한 설계

메모리 부담 없는 깊은 페이징

결과셋이 100건이든 50만 건이든, 요청한 페이지의 행만 엔티티가 됩니다. 맨 마지막 페이지에 도달해도 전체를 애플리케이션에 적재하지 않습니다.

LIMIT / OFFSET은 데이터베이스로 내려가고, 애플리케이션 계층은 페이지 크기만큼의 행만 매핑합니다. 메모리와 매핑 비용은 일정합니다 —— 1페이지든 5만 페이지든 O(페이지)이지 O(전체)가 아닙니다. ‘전부 메모리에 적재’하던 방식은 사라졌습니다.

동작 흐름

SQL에서 엔티티로, 그리고 다시 —— 자동으로

SQL을 쓰면 엔티티 목록을 얻습니다. 웹 폼은 표현 언어(EL)로 사용자 입력을 엔티티에 바인딩합니다. 중간 계층은 같은 엔티티에 saveOrUpdate만 호출하면 —— Modelet이 그것을 다시 SQL로 바꿉니다.

엔티티는 JPA 호환 애너테이션 —— @Table @Id @Column @Transient @Enumerated @GeneratedValue —— 으로 정의되어 Jakarta Persistence 엔티티와 똑같이 읽힙니다. 빌려온 것은 어휘뿐, 엔진은 처음부터 끝까지 SQL-first입니다.

빠른 시작

하나의 예제로 처음부터 끝까지

전형적인 일대다: Author는 여러 Book을 가집니다. 외래 키가 곧 관계입니다 —— @OneToMany도, 지연 로딩도 없습니다.

클래스 다이어그램

1JPA 호환 애너테이션으로 엔티티 정의

models.pyPython 3.10+
@Table("author")
@dataclass
class Author(Entity):
    name: str | None = None

@Table("book")
@dataclass
class Book(Entity):
    author_id: int | None = None          # FK = the relationship
    title: str | None = Column("bookName")
    price: Decimal | None = None

2아는 SQL로 조회

query.pyfind → list
# plain ANSI SQL in, a list of entities out
books = model.find(
    "select * from book where author_id = ? order by id",
    [author_id], Book)

3saveOrUpdate —— 프레임워크가 SQL을 작성

save.pysave
book = Book(author_id=author.id,
            title="pymodelet in Action")
model.save(book)    # new entity → INSERT, id written back

book.price = Decimal("499")
model.save(book)    # loaded entity → UPDATE. same call.
