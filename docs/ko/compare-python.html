pymodelet vs SQL-first 진영

Python 생태계의 모든 SQL-first 도구는 읽기 측을 챙겨 줍니다. pymodelet은 쓰기까지 편안하게 만들어 줍니다.

SQL을 실행해 list of entity를 받고, 그 list를 그대로 저장하면 추가·수정·삭제가 자동으로 판별됩니다 —— 현재 이것이 가능한 패키지는 pymodelet뿐입니다.

그럼에도 pymodelet이 다른 이유를 보입니다.

경쟁자들

네 가지 SQL-first 패키지 소개

pymodelet

읽기와 쓰기, 둘 다 답이 있다

순수 SQL을 넣으면 타입 있는 엔티티가 나옵니다 —— 어떤 DB-API 2.0 드라이버에서든, 의존성 제로로. 그리고 아무도 하지 않은 부분: saveOrUpdate가 엔티티로부터 모든 INSERT / UPDATE / DELETE를 생성하고, 키를 되써 주고, 감사 필드를 채웁니다.

psycopg 3 · row_factory

아름다운 읽기, 단 PostgreSQL 전용

class_row는 추가 패키지 없이 SELECT를 dataclass로 매핑합니다 —— PostgreSQL이라면 가장 깔끔한 읽기 경로. 하지만 쓰기 문장은 전부 직접 조립해야 하고, 키는 RETURNING을 기억해야만 돌아옵니다.

aiosql

SQL 파일이 함수가 된다

쿼리는 .sql 파일에 살고 호출 가능한 함수가 됩니다 —— 읽기 측 관리로는 가장 우아한 방식. 하지만 쓰기는 한 문장씩 손으로 쓰고, 돌아오는 것은 row 그대로: 타입 있는 엔티티로 바꾸는 일은 당신 몫입니다.

SQLAlchemy Core + Pydantic

배관은 탄탄, 매핑은 수동

검증된 풀 위에서 text() 쿼리를 돌리고, 엔티티는 Pydantic이 검증합니다. 하지만 매핑은 쿼리마다 수동 —— UserEntity(**row._mapping) —— 그리고 쓰기 측은 당신이 둔 그 자리에 그대로: 아직 안 써진 채로.

결정적 차이

pymodelet: 읽기도 쓰기도 간편, 코드는 더 읽기 쉽게, AI 토큰은 절약

흔한 데이터 CRUD 상황을 떠올려 보세요: 사용자를 등록해 저장하고, 나중에 이메일을 수정합니다. 다른 패키지는 두 SQL 문장을 직접 쓰게 하지만, pymodelet은 그것들을 사라지게 합니다 —— 당신과 AI는 비즈니스 로직에 집중하고, 무의미한 토큰 생성은 아낍니다.

aiosql — users.sql + Pythonevery statement, by hand
-- users.sql — you write and maintain each one
-- name: insert_user!
INSERT INTO users (name, email) VALUES (:name, :email);

-- name: update_user_email!
UPDATE users SET email = :email WHERE id = :id;

# Python
queries.insert_user(conn, name="Alice", email="alice@test.com")
queries.update_user_email(conn, id=1, email="alice@corp.com")
pymodeletzero statements
user = User(name="Alice", email="alice@test.com")
model.save(user)          # INSERT — id written back

user.email = "alice@corp.com"
model.save(user)          # UPDATE — same call, saveOrUpdate

# delete? user.txn_mode = TxnMode.DELETE; model.save(user)

컬럼 하나를 추가하면: aiosql에서는 테이블, INSERT, UPDATE, 그리고 필드를 나열하는 모든 호출부에 손이 갑니다. pymodelet은 테이블과 dataclass 필드 하나뿐 —— SQL은 스스로 다시 자랍니다.
