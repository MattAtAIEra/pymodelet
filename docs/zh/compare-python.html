pymodelet vs SQL-first 陣營

Python 生態系的每一個 SQL-first 工具都照顧到了讀取面。pymodelet 連寫入都讓你輕鬆自在。

pymodelet 是目前唯一讓你下 SQL 得到 list of entity,再把 list of entity 直接儲存,自動判斷新增、修改、刪除的套件。

以下是 pymodelet 依然與眾不同的地方。

競爭者們

四個 SQL-first 套件簡介

pymodelet

讀與寫,都有答案

純 SQL 進、型別化 entity 出 —— 站在任何 DB-API 2.0 驅動上,零依賴。還有別人都沒做的那一塊:saveOrUpdate 從 entity 自動產生每一句 INSERT / UPDATE / DELETE,回填主鍵、蓋上稽核欄位。

psycopg 3 · row_factory

漂亮的讀取,僅限 PostgreSQL

class_row 把 SELECT 直接映射成 dataclass,一個額外套件都不用 —— 若你用 PostgreSQL,這是最乾淨的讀取路徑。但每一句寫入語句仍要自己組,主鍵也得記得寫 RETURNING 才拿得回來。

aiosql

SQL 檔案即函式

查詢住在 .sql 檔裡、變成可呼叫的函式 —— 管理讀取面最優雅的方式。但寫入語句要一句一句手寫,查回來的是 row 不是 entity:轉型是你的工作。

SQLAlchemy Core + Pydantic

管線紮實,映射手動

text() 查詢跑在久經沙場的連線池上,entity 交給 Pydantic 驗證。但映射是每條查詢手動一次 —— UserEntity(**row._mapping) —— 寫入面則原封不動:還沒寫。

pymodelet 讓讀寫都便利,程式更易讀,AI 省 token

假設一個資料 AUDI 情境:新增一位使用者、存檔,後來又修正 email。其他套件都要你把兩句 SQL 寫出來;pymodelet 讓它們直接消失。真正讓你跟 AI 專注商業邏輯,節省無謂的 token 產出。

aiosql — users.sql + Pythonevery statement, by hand
-- users.sql — you write and maintain each one
-- name: insert_user!
INSERT INTO users (name, email) VALUES (:name, :email);

-- name: update_user_email!
UPDATE users SET email = :email WHERE id = :id;

# Python
queries.insert_user(conn, name="Alice", email="alice@test.com")
queries.update_user_email(conn, id=1, email="alice@corp.com")
pymodeletzero statements
user = User(name="Alice", email="alice@test.com")
model.save(user)          # INSERT — id written back

user.email = "alice@corp.com"
model.save(user)          # UPDATE — same call, saveOrUpdate

# delete? user.txn_mode = TxnMode.DELETE; model.save(user)

加一個欄位:在 aiosql 要動資料表、INSERT、UPDATE,還有每個點名欄位的呼叫處。在 pymodelet 只要動資料表和一個 dataclass 欄位 —— SQL 自己重新長出來。
